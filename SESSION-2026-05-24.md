### Track 5 — SBM v2.2.0 release (commit `09df4c4`, tag `v2.2.0`)

First schema bump since v2.0. **Additive only** — every v2.0-conforming
model is also v2.2-conforming.

- New `telemetry_stream` entity type (28th): time-series sensor data
  with `dataReference` (file_csv / bacnet / mqtt / api / lorawan),
  `summaryStatistics` (min/max/mean + rolling means + thresholds with
  `currentlyExceeded` + `exceededSince`), `qualityMetadata` (coverage,
  outage events, calibration due). Pattern `^TEL-[A-Z0-9-]+$`.
- New `asset.operationalHistory` typed sub-schema: `runtimeHours`,
  `cycleCount`, `energyConsumed_kWh`, `servicedAt` array (date, type
  enum, technician, findings, cost, relatedIssueId), `performanceTrend`.
- 2 telemetry_stream entities retrofitted into 2028 example
  (`TEL-CO2-402-001` + `TEL-HP-COP-001`).
- Schema file `schemas/sbm-schema-v2.2.json` (v2.0 stays frozen).
- Compiler `VERSION` 2.1 → 2.2; `sbm_version` emitted as "2.2".

## Working method this session

- **User-driven escalation**: each phase the user said "do all" or
  "continue", trusting the agent to keep building. 13 commits in one
  session, ranging from a 1-file compiler fix to a 28-entity schema bump.
- **Test → fix → ship loop kept tight**: `npm test` + the 3 example
  `sbm:validate` commands run after most material changes, before
  committing.
- **Schema discipline**: v2.0 stayed frozen when v2.2 shipped; v2.2
  stayed frozen when v2.3 shipped. New schema files, not edits.
- **Examples drove schema design**, not the other way round: 2028
  example's `SCHEMA-GAPS.md` was authored first as input to v2.2/v2.3
  design.
- **Bilingual parity maintained throughout** — every new entity in
  the 3 examples has EN + PL versions; every i18n-aware target
  (html-report, diagrams) updated in both locales.
